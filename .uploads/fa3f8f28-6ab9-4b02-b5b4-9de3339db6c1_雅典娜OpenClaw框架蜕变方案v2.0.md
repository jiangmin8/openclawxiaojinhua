# 雅典娜 OpenClaw 框架蜕变方案 v2.0

> **版本：** 2.0
> **日期：** 2026-05-15
> **目标环境：** WSL2 Linux (systemd)，OpenClaw 运行时宿主
> **运行模型：** DeepSeek V4 Pro
> **核心原则：** 边界感 + 网络安全 + 最小权限 + 可落地执行 + 零代码门槛

---

## 摘要

本方案基于对雅典娜系统现有 7 个核心 MD 文件 + 1 个 PowerShell 脚本 + 1 份历史审计报告的全面诊断，提出**三阶段蜕变路径**：

1. **骨骼重塑** — 消除冗余、统一格式、修复缺陷，建立清晰的文件职责边界
2. **血肉填充** — 初始化知识库、填充上下文、编写自动化脚本，让框架"能跑"
3. **铠甲锻造** — 强化网络安全、加固权限边界、建立自检机制，确保系统安全自洽

蜕变完成后，雅典娜将从"文档雏形"进化为**可自主运行、自检自愈、安全可控的智能执行系统**。

## 1. 现状诊断

### 1.1 文件体系全景

| 文件 | 当前状态 | 核心问题 |
|:---|:---|:---|
| `IDENTITY.md` | ✅ 基本可用 | 缺少版本号；"头像"资源未创建；协议依存表缺少 `agent.md` 和 `CHANGELOG.md` |
| `SOUL.md` | ⚠️ 严重污染 | 底部残留 ~1000 行历史对话记录（含思考过程、多轮对话），严重污染系统提示词，浪费 Token |
| `AGENTS.md` | ✅ 结构良好 | v3.1 版本号与审计报告 v3.0 不一致；术语定义仅提及 OpenClaw 但未定义其架构 |
| `MEMORY.md` | ⚠️ 空壳 | 5 个子文件全部"待创建"；模板路径与 AGENTS.md 实际路径不一致 |
| `TOOLS.md` | ✅ 基本可用 | 与 AGENTS.md 存在职责重叠（两者都定义了"输出标准"和"异常处理"） |
| `USER.md` | ✅ 基本可用 | "意图包"概念未在其他文件中引用；缺少优先级规则（审计报告中补充了但未合并） |
| `HEARTBEAT.md` | ⚠️ 部分失效 | API 连通性检查引用了未定义的 `{API_ENDPOINT}` 和 `{proxy}` 占位符；路由器 IP 引用 `MEMORY.md 子文件` 但该文件不存在 |
| `agent.md` | ⚠️ 职责混乱 | 与 AGENTS.md 高度重叠；路径仍硬编码为旧 Windows 路径 `C:\Users\lz\Desktop\12` |
| `context.md` | ❌ 完全空白 | 项目名、目标、阶段全部未填写，仅保留模板占位符 |
| `CHANGELOG.md` | ❌ 几乎为空 | 仅一条记录，无法追溯系统演进历史 |
| `snapshot-context.ps1` | ⚠️ 路径失效 | 硬编码 `C:\Users\lz\Desktop\12`，与 WSL2 环境不兼容 |

### 1.2 架构级问题

#### 问题 1：文件职责边界模糊

```
当前混乱关系：
  AGENTS.md ←→ agent.md   （两者都定义执行规范，内容重叠 60%+）
  TOOLS.md  ←→ AGENTS.md  （两者都定义异常处理和输出标准）
  SOUL.md   ←→ USER.md    （两者都定义沟通协议和输出格式）
```

**影响：** 雅典娜在执行时无法确定应遵循哪个文件的规则，可能导致行为不一致。

#### 问题 2：知识库完全为空

MEMORY.md 声明了 5 个子文件，但全部"待创建"。这意味着：
- 雅典娜不知道自己的运行环境（IP、端口、硬件配置）
- 雅典娜不知道当前项目上下文
- 雅典娜不知道大哥的偏好
- HEARTBEAT.md 引用的配置项（API 地址、代理、网关 IP）无处可查

#### 问题 3：安全机制存在缺口

| 缺口 | 描述 | 风险等级 |
|:---|:---|:---|
| 无网络白名单 | TOOLS.md 提到"白名单域名"但未在任何文件中定义具体白名单 | 高 |
| 无文件完整性校验 | 核心文件被篡改后无检测机制 | 高 |
| 无会话隔离 | 多会话并发时无上下文隔离机制 | 中 |
| API 密钥管理粗糙 | 仅说"存储在 .env"，无加密、无轮换策略 | 中 |
| 审计日志未实现 | agent.md 定义了审计格式但无脚本生成审计日志 | 中 |

#### 问题 4：自动化能力为零

- HEARTBEAT.md 定义了守护任务，但无实际脚本执行
- snapshot-context.ps1 是 PowerShell 脚本，不兼容 WSL2
- 无启动自检脚本
- 无知识库初始化脚本

### 3.2 自动化脚本体系

#### 3.2.1 `scripts/athena-boot.sh` — 启动自检脚本

**功能：**
1. 检查所有核心 MD 文件是否存在
2. 验证核心文件 SHA256 完整性指纹
3. 检查 OpenClaw Gateway 服务状态
4. 检查 API 连通性
5. 检查磁盘空间
6. 加载 MEMORY.md 索引，报告知识库状态
7. 输出启动报告

**执行时机：** 每次雅典娜会话启动时自动执行

#### 3.2.2 `scripts/athena-heartbeat.sh` — 守护任务执行器

**功能：**
1. 高频检查（每小时）：API 连通性、Gateway 保活、路由器连通性
2. 每日任务（22:00）：备份完整性校验、日志清理
3. 系统检查（每6小时）：资源监控、性能报告生成
4. 告警通知（仅 P0/P1 级别推送给大哥）

**执行方式：** 通过 `cron` 或 OpenClaw 内置定时器调度

#### 3.2.3 `scripts/athena-backup.sh` — 快照备份脚本

**功能：**
1. 备份所有核心 MD 文件至 `/mnt/g/backup/athena/{timestamp}/`
2. 生成 SHA256 校验文件
3. 写入审计日志
4. 清理超过 30 天的旧备份

**替代：** `snapshot-context.ps1` 的 Bash 版本

#### 3.2.4 `scripts/athena-init-memory.sh` — 知识库初始化脚本

**功能：**
1. 创建 `memory/` 目录结构
2. 基于系统命令自动采集环境信息，生成 `context.base.md` 初版
3. 创建空模板文件（`context.project.md`、`context.user.pref.md` 等）
4. 输出待大哥手动填充的字段清单

**执行时机：** 首次部署时执行一次

#### 3.2.5 `scripts/athena-security-audit.sh` — 安全自检脚本

**功能：**
1. 检查核心文件完整性（对比 SHA256 指纹）
2. 检查 `.env` 文件权限（应为 600）
3. 检查是否有新增的未知文件
4. 扫描审计日志中的异常记录
5. 输出安全报告

**执行时机：** 每日 03:00 自动执行

## 4. 蜕变方案 — 第三阶段：铠甲锻造

### 4.1 网络安全强化

#### 4.1.1 网络白名单机制

在 `TOOLS.md` 中定义三级网络白名单：

| 等级 | 域名示例 | 说明 |
|:---|:---|:---|
| **信任域** | `*.github.com`, `*.openwrt.org`, `pypi.org`, `npmjs.org` | 可自动访问，无需确认 |
| **受限域** | `huggingface.co`, `*.docker.com` | 可下载但需记录日志 |
| **禁止域** | `*.onion`, 已知恶意 IP 段 | 硬拒绝，触发安全告警 |

**实现方式：** 在 `.env` 中配置 `NETWORK_WHITELIST_TRUSTED` 和 `NETWORK_WHITELIST_RESTRICTED` 变量。

#### 4.1.2 API 密钥安全加固

| 措施 | 说明 |
|:---|:---|
| 权限隔离 | `.env` 文件权限设为 `600`（仅所有者可读写） |
| 禁止回显 | TOOLS.md 中明确禁止在日志、输出、对话中包含 API Key 的任何部分 |
| 熔断机制 | 429/5xx 时停止重试，60 秒冷却，最多 3 次 |
| 成本预警 | 单次操作预估超过 8000 Token 时需大哥确认 |
| 密钥轮换 | 建议每 90 天轮换一次 API Key |

#### 4.1.3 端口暴露防护

- 所有服务仅监听 `127.0.0.1`（localhost）
- 禁止使用 `0.0.0.0` 绑定
- Docker 容器端口映射仅使用 `127.0.0.1:{host}:{container}`
- 定期扫描监听端口（`ss -tlnp`），发现异常立即告警

### 4.2 文件完整性保护

#### 4.2.1 核心文件指纹机制

在 `.security/checksums.sha256` 中存储所有核心 MD 文件的 SHA256 哈希值：

```
# 雅典娜核心文件完整性指纹
# 生成命令: sha256sum IDENTITY.md SOUL.md AGENTS.md TOOLS.md USER.md HEARTBEAT.md MEMORY.md CHANGELOG.md
# 最后更新: 2026-05-15

abc123...  IDENTITY.md
def456...  SOUL.md
ghi789...  AGENTS.md
...
```

**校验时机：**
- 每次启动自检（`athena-boot.sh`）
- 每日安全审计（`athena-security-audit.sh`）
- 核心文件修改后自动更新指纹

**异常处理：**
- 指纹不匹配 → 触发 `[CRITICAL SECURITY ALERT]`
- 暂停所有 Tier 2/3 操作
- 通知大哥，等待人工裁决

#### 4.2.2 变更追踪

所有核心文件的修改必须：
1. 修改前自动备份至 `.backups/{timestamp}/`
2. 修改后更新 `checksums.sha256`
3. 在 `CHANGELOG.md` 中追加记录
4. 在 `memory/logs/operation.log` 中写入审计条目

### 4.3 权限边界强化

#### 4.3.1 写入边界（硬编码）

在 `AGENTS.md` 中以代码级确定性定义路径白名单：

```bash
# 允许写入的路径（正则匹配）
ALLOWED_WRITE_PATHS=(
  "^/mnt/d/"
  "^/mnt/e/"
  "^/mnt/f/"
  "^/mnt/g/"
  "^/home/lz/"
  "^/tmp/"
)

# 严格禁止的路径
FORBIDDEN_PATHS=(
  "/mnt/c/"
  "/etc/"
  "/usr/"
  "/boot/"
  "/sys/"
  "/proc/"
  "/root/"
)
```

#### 4.3.2 命令执行黑名单

在 `TOOLS.md` 中定义高危命令模式（正则匹配）：

```
FORBIDDEN_PATTERNS=(
  "sudo\s+rm"           # sudo 删除
  "rm\s+-rf\s+/"        # 递归强制删除根目录
  "chmod\s+777"         # 开放全部权限
  "dd\s+if="            # 磁盘直写
  "mkfs\."              # 格式化文件系统
  "> /dev/sd"           # 直接写块设备
  "curl.*\|\s*bash"     # 远程脚本执行
  "wget.*\|\s*sh"       # 远程脚本执行
)
```

### 4.4 审计体系

#### 4.4.1 审计日志格式

所有 Tier 2/3 操作必须记录审计日志，格式：

```
[ISO8601_TIMESTAMP] | [ACTOR] | [ACTION] | [TARGET] | [TIER] | [RESULT] | [USER_AUTH]
```

示例：
```
2026-05-15T14:30:00+08:00 | athena | fs_write | /mnt/e/Projects/config.yaml | T2 | success | auto
2026-05-15T14:35:00+08:00 | athena | fs_delete | /tmp/old_cache/ | T3 | success | user_confirmed
```

#### 4.4.2 审计日志管理

| 规则 | 说明 |
|:---|:---|
| 存储路径 | `memory/logs/operation.log` |
| 轮转策略 | 单文件超过 10MB 时自动轮转（保留最近 5 个） |
| 保留期限 | 90 天自动清理 |
| 异常检测 | 每日扫描连续失败记录，超过 5 次触发告警 |
