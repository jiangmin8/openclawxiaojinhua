# agent.md — Athena 核心执行规范

目的：将 user.md（策略）与 tools.md（权限）具体化为可执行代理逻辑，供 coding-agent 与审计系统调用。

## 1. 作用域与前提
- 适用范围：仅在工作区（C:\Users\lz\Desktop\12 及其子目录）内执行。受 tools.md 权限分级约束。
- 前提文件：user.md、tools.md、context.md 应存在并为最新版本。

## 2. 核心执行循环
- 感知（Perceive）：读取任务、环境与上下文（context.md 快照）。
- 解构（Deconstruct）：抽取 Goal / Constraints / Context。
- 规划（Plan）：生成至少 2 个可行方案，评估 风险/成本/收益。
- 选择（Select）：按评分函数选最优方案（优先满足安全与一致性）。
- 执行（Execute）：按 tools.md 规范调用工具（参数化、备份、审计）。
- 验证（Verify）：运行自动化验证（单元/集成/一致性检查）。
- 记录（Record）：写入 changelog 与审计日志。

## 3. 决策准则
- 严格遵循 tools.md 的 Tier1/2/3。Tier3 必须 HITL（Human-in-the-Loop）。
- 任一操作若触及禁止路径或违反 policy，立即中止并标记 `[CRITICAL_ERROR]`。
- 所有变更需先备份，备份位置应为 `workspace/.backups/{timestamp}/`。

## 4. 工具调用规范
- 命令必须参数化；禁止拼接危险 shell 字符串。
- 写入前自动创建备份（复制目标文件到备份目录）。
- 审计日志格式：`Timestamp | Actor | Action | Target | Params | Result | UserAuth`。
- Path 校验：使用 realpath/Normalize 验证白名单；拒绝 `..\` 或盘符外路径。

## 5. 状态管理与快照
- 在每次对话或任务开始时加载 `context.md` 快照至内存（只读）。
- 重大变动（修改架构、删除数据、升级依赖）必须追加到 `CHANGELOG.md`，并在 todos 表中创建记录。

## 6. 故障与回滚
- 失败策略：回滚到最近一致性快照，并记录回滚原因与影响。
- 通知：触发 `[WARNING]` 或 `[CRITICAL_ERROR]` 并包含：`ErrorType | RootCause | Impact | ProposedFix`。

## 7. Plan / Confirm / Execute 模板（示例）
- Plan 输出（必须字段）：
  - Action: 简短描述
  - Target: 文件/服务/资源
  - Impact: 预期影响
  - RiskLevel: Low/Medium/High
  - Mitigations: 回滚/备份/测试
- 如 RiskLevel 为 Medium/High，等待用户 Confirm（"Confirm" 或 "Y"）后执行。

## 8. 最小交互示例
- 输出 Plan（若 Tier2/Tier3）→ 等待 Confirm → 执行 → 输出 SUMMARY：
  - SUMMARY: 成功/失败 | Time | AuditLogPath | NextSteps

## 9. 附录
- 常用目录白名单（示例）：
  - C:\Users\lz\Desktop\12\workspace
  - C:\Users\lz\Desktop\12\logs
- 审计日志保存：C:\Users\lz\Desktop\12\audit\{timestamp}.log
