## 1. 安装/配置总序与路径规范

### 1.1 核心路径与扩展路径定义

#### 1.1.1 `~/.openclaw` — 核心配置文件目录（SOUL.md, USER.md, MEMORY.md）

`~/.openclaw` 是 OpenClaw 系统的配置中枢，承载 AI 人格定义、用户画像绑定与记忆持久化等核心职能。该目录位于用户主目录下，采用隐藏文件夹设计，符合 Linux/Unix 系统的配置惯例，确保核心文件不会被误操作或意外删除。根据 OpenClaw 官方文档的架构说明，此路径在首次运行 `openclaw onboard` 时自动初始化，但手动预创建目录骨架可规避权限问题，确保后续配置流程的顺畅性。

该目录下的关键文件包括：
- `SOUL.md`：AI 底层人格定义，决定行为模式与沟通风格
- `USER.md`：用户画像绑定，记录硬件指纹、工作偏好与沟通禁忌
- `MEMORY.md`：长期记忆索引，指向三层记忆架构的具体位置
- `openclaw.json`：全局配置文件，定义模型供应商、API 密钥、网关端口等参数
- `memory/`：自动生成的向量记忆库目录，含 SQLite 索引与嵌入文件
- `sessions/`：会话日志存储，支持跨会话的状态恢复

验证该目录存在性与权限合规的标准命令：

```bash
# 核心路径验证
ls -la ~/.openclaw 2>/dev/null || echo "DIR_MISSING: run mkdir -p ~/.openclaw"

# 深度结构探测
find ~/.openclaw -maxdepth 2 -type d 2>/dev/null | sort

# 权限确认（OpenClaw 进程通常以当前用户运行）
stat -c "%U:%G %a %n" ~/.openclaw 2>/dev/null
```

建议设置 `chmod 700 ~/.openclaw` 限制访问权限，因为该目录可能包含 API Key、SSH 私钥引用等敏感信息。对于 WSL2 环境，需特别注意 Windows 文件系统权限映射问题——若 `stat` 输出显示 `root:root` 或 `777` 权限，表明存在权限继承异常，需手动修正以避免 OpenClaw 拒绝加载配置文件。

#### 1.1.2 `~/openclaw` — 项目代码及扩展目录（Agents/, Skills/）

`~/openclaw` 作为功能扩展的根节点，承载多智能体（Agents）分工体系与技能插件（Skills）的动态扩展功能。与核心路径的隐藏属性不同，该目录采用显式命名，便于用户频繁访问、版本控制与团队协作。根据官方文档的加载优先级说明，工作区级别的 Skills（即 `~/openclaw/skills/`）具有最高优先级，可覆盖内置技能与个人级技能，为场景定制化提供极大灵活性。

该目录的标准结构为：
- `agents/`：多智能体定义目录，每个子目录对应一个专业 Agent
- `skills/`：技能插件目录，支持从 ClawHub 市场安装或本地自定义开发
- `scripts/`：用户自定义脚本，如记忆归档自动化脚本

## 2. AI 底层人格设定（SOUL.md）

### 2.1 文件定位与创建命令

#### 2.1.1 路径：`~/.openclaw/SOUL.md`

`SOUL.md` 是 OpenClaw 系统中权重最高的单文件配置，其内容在每次对话请求时作为系统提示（system prompt）注入上下文窗口，直接覆盖模型的默认行为模式。该文件位于 `~/.openclaw/SOUL.md`，采用 Markdown 格式，通过结构化的 frontmatter 与自由文本段落相结合的方式，实现对 AI 人格的多维度定义。官方文档强调，SOUL.md 的修改应极为审慎，频繁变动会导致 AI 行为的不一致性，因此建议在初次配置时投入充分时间进行精细化打磨，后续仅做微调。

创建与验证的完整命令：

```bash
# 创建并写入（使用 here-document 避免转义问题）
cat > ~/.openclaw/SOUL.md << 'SOUL_EOF'
[完整内容见 2.2-2.4 节]
SOUL_EOF

# 验证：检查文件存在、非空、首行匹配预期
test -s ~/.openclaw/SOUL.md && \

`test -s` 确保文件大小大于零，`head -1 | grep -q "name:"` 验证 YAML frontmatter 格式正确。若验证失败，需检查 heredoc 的定界符是否被意外转义，或文件系统是否以只读模式挂载。

### 2.3 性格与沟通风格

"直接急躁"的沟通风格需要通过具体的行为原则来量化定义，而非停留在抽象的形容词层面。有效的风格定义应当包含正向指令（"做什么"）和负向指令（"不做什么"）的对称结构。

- **分段输出禁令**：禁止将长命令拆分为多段发送，若命令长度超过终端宽度，应使用反斜杠 `\` 换行保持单条命令的完整性

**技术导向：WSL2 Ubuntu / Windows 11 双环境认知**

### 3.2 硬件与环境指纹

用户画像的硬件层需要精确到影响命令生成的技术参数。A5-10400 作为 AMD Zen2 架构的 OEM 处理器，其 6 核心 6 线程（无 SMT）设计意味着并行任务上限为 6，Docker 的 `--cpus` 参数或 `make -j` 的线程数需据此调整。16GB RAM 在 WSL2 的默认 50% 内存分配策略下，实际可用约 8GB，这决定了 Docker 容器的 `--memory` 限制必须预留 2GB 给 Windows 宿主，避免触发 WSL2 的 OOM killer 导致整个虚拟机崩溃。GT730 的 GK208 核心基于 Kepler 架构，计算能力 3.5，不支持 CUDA 10.0+ 之后的 runtime，这意味着任何涉及 GPU 加速的建议（如 PyTorch 的 `cuda()` 调用）必须被自动拦截并替换为 CPU 模式或 ROCm 替代方案（若未来升级 AMD 显卡）。

网络拓扑的双路由架构是画像的关键差异化信息：GL.iNet MT3600BE 作为主路由承担 DHCP、防火墙、VPN 等核心功能，其 OpenWrt 固件允许深度定制；小米 AX3000 作为 Mesh 节点或 AP，负责无线覆盖与 IoT 设备隔离。这一拓扑决定了网络诊断命令必须区分执行目标（`ssh root@192.168.8.1` 为 MT3600BE 默认地址，`ssh root@192.168.31.1` 为小米路由器默认地址），且 VLAN 划分策略需考虑两设备间的 trunk 兼容性。

## 4. 三层记忆架构设计

### 4.1 目录结构与自动化规则

OpenClaw 的记忆体系采用显式的 Markdown 文件作为真相源（source of truth），区别于传统向量数据库的黑盒存储。这种设计使得人类用户可以直接编辑记忆内容，审计 AI 的学习过程。三层架构在时间维度上形成漏斗：短期记忆保留完整对话，中期提炼提取模式与决策，长期知识库存储可复用的领域知识。

| 层级 | 物理路径 | 文件格式 | 内容类型 | 生命周期 | 触发写入 |
|:---|:---|:---|:---|:---|:---|

自动化归档脚本框架（`~/openclaw/scripts/memory/archive.sh`）：

```bash
#!/bin/bash
# memory-archive.sh — 三层记忆自动化归档
# 执行频率: 每日 00:00 (short_term 滚动) + 每周日 00:00 (medium_term 提炼) + 每月末 (long_term 归档)

set -euo pipefail

MEMORY_ROOT="${HOME}/.openclaw/memory"
TODAY=$(date +%Y-%m-%d)
WEEK=$(date +%Y-W%V)
MONTH=$(date +%Y-%m)

# --- 短期记忆滚动 ---
SHORT_TERM="${MEMORY_ROOT}/short_term"
find "${SHORT_TERM}" -name "*.md" -mtime +7 -type f | while read -r f; do
    # 压缩至 archive 并删除原文件
    tar -czf "${MEMORY_ROOT}/archive/$(basename "$f").tar.gz" -C "${SHORT_TERM}" "$(basename "$f")"
    rm -f "$f"
done

# --- 中期提炼（每周日触发） ---
if [[ "$(date +%u)" == "7" ]]; then
    WEEKLY_FILE="${MEMORY_ROOT}/medium_term/weekly-${WEEK}.md"
    # 调用 OpenClaw API 或本地 LLM 进行提炼（简化版：基于 grep 统计）
    cat > "${WEEKLY_FILE}" << EOF
---
week: ${WEEK}
generated: ${TODAY}
---

## 高频命令（自动统计）
$(grep -h "^- \[.*\] \`" "${SHORT_TERM}"/*.md 2>/dev/null | sed 's/.*`\([^`]*\)`/\1/' | sort | uniq -c | sort -rn | head -10 | awk '{print "- " $2 " " $1 "次"}')

## 用户纠正汇总
$(grep -A2 "用户纠正" "${SHORT_TERM}"/*.md 2>/dev/null | grep "^- " | sort | uniq -c | sort -rn | head -5)
EOF
fi

# --- 长期归档（每月末触发） ---
LAST_DAY_OF_MONTH=$(date -d "$(date +%Y-%m-01) +1 month -1 day" +%d)
if [[ "$(date +%d)" == "${LAST_DAY_OF_MONTH}" ]]; then
    # 将本月 medium_term 合并至 long_term 对应 topic
    # 简化实现：基于 tags 聚合，实际应由 LLM 进行语义分类
    for topic in docker network router python; do
        KNOWLEDGE_FILE="${MEMORY_ROOT}/long_term/knowledge-${topic}.md"
        # 追加或创建
        grep -l "${topic}" "${MEMORY_ROOT}/medium_term"/weekly-*.md 2>/dev/null | head -5 | xargs cat >> "${KNOWLEDGE_FILE}.tmp"
    done
fi

# 容量检查与告警
CURRENT_SIZE=$(du -sh "${MEMORY_ROOT}" | awk '{print $1}')
echo "Archive complete. Current storage: ${CURRENT_SIZE}"
```

### 5.2 路由器管理 Agent（router-admin）

**角色定义**：专精 OpenWrt/GL.iNet 固件与小米路由器生态的网络设备管理员。区别于通用网络诊断工具，该 Agent 需深度理解 uci 配置体系的 transactional 特性（`uci set` 累积变更，`uci commit` 原子提交，`uci revert` 回滚），以及小米官方固件的封闭性限制（SSH 需开发者模式解锁，mimesh 命令集无文档）。

**专属 Prompt 模板**：

```markdown

# 继承主 SOUL.md 全部规则，追加以下专属约束

## 执行环境强制标注
- 所有命令前缀必须包含目标设备标识：
  - MT3600BE: `[MT3600BE@192.168.8.1]`
  - AX3000: `[AX3000@192.168.31.1]`
- SSH 命令必须使用 `-o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null` 避免首次连接交互阻塞

## uci 操作原子性规则
1. 修改前强制备份：`ssh root@$IP "uci export network > /tmp/network.bak.$(date +%s)"`
2. 批量变更使用 heredoc 减少 SSH 往返：
   ```bash
   ssh root@192.168.8.1 << 'EOF'
   uci set network.lan.ipaddr='192.168.8.1'
   uci set network.lan.netmask='255.255.255.0'
   uci commit network
   /etc/init.d/network restart
   EOF
   ```
3. 回滚指令必须随附：`ssh root@$IP "uci revert network; /etc/init.d/network restart"`

## 固件操作红线
- 禁止自动执行 `sysupgrade -n`（清除配置），必须显式保留配置：`-c /etc/sysupgrade.conf`
- 升级前强制检查 `/overlay` 剩余空间：`df -h /overlay | awk 'NR==2 {print $4}'`
- 小米设备禁止建议第三方固件刷入（变砖风险高，无 TTL 救砖接口）

## 网络诊断标准流程
1. 链路层：`ethtool eth0` 或 `swconfig dev switch0 show`
2. IP 层：`ip addr show; ip route show`
3. 传输层：`netstat -tulnp | grep :$PORT` 或 `ss -tulnp`
4. 应用层：`logread | tail -50` 或 `dmesg | grep -i error`
5. 无线层：`iwinfo wlan0 info` 或 `iw dev wlan0 station dump`
```

### 5.3 代码执行 Agent（code-runner）

**角色定义**：WSL2/Linux 环境的命令代理执行者，专注于 bash 脚本、Python 程序、Docker 操作的自动化运行。该 Agent 是用户"直接输出完整命令"偏好的主要承载体，需具备命令链的容错设计能力（如 `set -euo pipefail` 的自动注入）以及跨系统路径转换的隐式处理（`wslpath` 的自动调用）。

**专属 Prompt 模板**：

```markdown

## 命令生成强制规范
1. ** shebang 强制**：所有脚本必须以 `#!/bin/bash` 或 `#!/usr/bin/env python3` 开头，禁止裸命令序列
2. ** 严格模式自动注入**：bash 脚本第二行必须为 `set -euo pipefail`（错误退出、未定义变量报错、管道失败传播）

- `systemctl` 操作的是 WSL2 实例，非 Windows 宿主；systemd 未启用时回退到 `service` 命令

**角色定义**：系统级故障的诊断与修复专家，覆盖 Linux 系统错误、网络连通性问题、服务启动失败、资源耗尽等场景。核心方法论为：日志驱动（log-driven），即从 `journalctl`、`dmesg`、应用日志出发定位根因，而非猜测。

**专属 Prompt 模板**：

```markdown

# 继承主 SOUL.md 全部规则，追加以下专属约束

## 诊断流程（强制顺序）
1. **信息收集**（不可跳过）：
   - 系统日志：`journalctl -xe --since "10 minutes ago" | tail -50`
   - 内核日志：`dmesg | tail -50`
   - 资源状态：`free -h && df -h && uptime`
   - 服务状态：`systemctl list-units --failed`
   - 网络状态：`ss -tulnp && ip route`

2. **假设生成**（Top-3 排序）：
   - 基于日志模式匹配生成概率最高的三个根因假设
   - 每个假设标注置信度（high/medium/low）和验证命令

3. **快速验证**（最小化测试）：
   - 设计单变量验证实验，隔离其他因素干扰
   - 优先选择非破坏性验证（只读查询、模拟执行）

4. **根因确认**（排除其他可能性）：
   - 验证通过后，明确声明"已排除假设 B、C，确认根因为 A"
   - 验证失败时，返回步骤 2 生成新假设

5. **修复执行**（含回滚方案）：
   - 每个修复命令附带 undo 指令
   - 破坏性修复前自动创建系统快照或配置备份

6. **验证闭环**（确认修复）：
   - 修复后重新执行步骤 1 的收集命令，对比前后差异
   - 输出 "RESOLVED" 或 "PARTIAL（需进一步处理）"

## 输出格式强制规范
```markdown
## 根因: [一句话结论，禁止模糊表述]

## 证据链
| 来源 | 关键片段 | 时间戳 |
|:---|:---|:---|
| journalctl | `[日志内容]` | [时间] |
| dmesg | `[日志内容]` | [时间] |

## 修复命令
```bash
[完整命令]
```
**回滚**: `[回滚命令]`

## 验证结果
[修复后的状态对比]
```
```

### 7.1 日常调教标准操作流程（SOP）

OpenClaw 的持续优化依赖"即时纠正→偏好固化→记忆归档"的闭环。官方机制中，AI 模型自身会判断用户偏好并主动写入 USER.md 或 MEMORY.md，但用户可通过 SOUL.md 引导更主动的记录行为。

```markdown
# ~/.openclaw/.tuning_log 格式（追加写入）

session_id: sess_20260429_1430
type: [style|fact|permission|omission]

3. 执行 `read` → `edit` 操作更新文件

### 7.2 每周优化检查清单

建议每周日 02:00 执行（与中期记忆提炼同步），通过 `sys-maintainer` Agent 或手动脚本完成：

| 检查层级 | 检查项 | 通过标准 | 修复动作 |
|:---|:---|:---|:---|
| **文件层** | SOUL.md 稳定性 | 本月无变更，或变更已标注版本 | 如有漂移，对比 git diff |
| | USER.md 时效性 | 硬件/环境信息与实际一致 | 更新并标注日期 |

| **记忆层** | short_term 膨胀 | 单文件 < 500 行，总大小 < 10MB | 触发归档脚本 |
| | medium_term 提炼质量 | 每周文件存在且含结构化摘要 | 如无，手动执行提炼 |
| | long_term 检索效率 | `memory_search` 返回 < 3 秒 | 重建向量索引 |
