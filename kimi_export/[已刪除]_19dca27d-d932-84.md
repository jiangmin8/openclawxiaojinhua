针对 **本地 Ollama 版 Hermes Agent**，日常任务要围绕**模型生命周期管理、本地资源优化、Ollama 服务稳定性**来设计。以下是精简实用的清单：

---

## 一、Ollama 服务层面

| 任务 | 具体操作 | 频率 |
|------|---------|------|
| **服务存活检查** | `curl http://localhost:11434/api/tags` 测试响应 | 每 15 分钟 |
| **进程资源监控** | 检查 `ollama` 进程 CPU/内存，警惕内存泄漏（长期运行后内存只增不减） | 每小时 |
| **模型热加载检查** | 确认常用模型（如 hermes3/gemma4）已在内存中，避免频繁冷启动拖慢响应 | 每 6 小时 |
| **版本更新检查** | `ollama -v` 对比官网最新版，评估是否升级 | 每周 |

**异常处理**：若 Ollama 无响应，先 `systemctl restart ollama`（或 Docker 重启），并记录重启原因。

---

## 二、模型文件管理

| 任务 | 具体操作 |
|------|---------|
| **磁盘占用审计** | `du -sh ~/.ollama/models/` 查看总量，模型文件很容易占几十 GB |
| **过期模型清理** | 删除长期未用的模型：`ollama rm <model_name>` |
| **模型完整性校验** | 对关键模型文件跑 `sha256sum` 比对官方 manifest |
| **新模型预拉取** | 如需更新 hermes3，提前 `ollama pull` 到本地，避免高峰期下载 |

**注意**：Ollama 的模型存储在 `~/.ollama/models/`，系统用户名都是 `lz`，所以路径是 `/home/lz/.ollama/models/`。

---

## 三、Agent 上下文与状态

| 任务 | 具体操作 |
|------|---------|
| **会话上下文清理** | 清理超过 24 小时无活动的会话，释放内存中的对话历史 |
| **长期记忆压缩** | 若 Agent 有记忆功能，对过期记忆做摘要归档，避免上下文窗口膨胀 |
| **失败任务重试** | 检查队列中因 Ollama 超时（如生成长文本时）失败的任务，适度重试 |
| **响应质量采样** | 随机抽查当日 5-10 条 Agent 输出，检查是否有模型退化迹象 |

给 Hermes Agent（或任何长期运行的 AI Agent）布置日常任务时，核心目标是**保持系统健康、数据新鲜、资源可控**。以下是建议的基础清理和检查清单：

---

## 一、日常清理任务

### 1. 日志管理
- **清理过期日志**：删除/归档 N 天前的运行日志（如 7 天或 30 天，视磁盘容量定）
- **日志轮转**：确保日志文件不会无限增长，设置单文件大小上限（如 100MB）
- **错误日志提取**：扫描当日 ERROR/FATAL 级别日志，汇总发送告警摘要

### 2. 临时文件清理
- **清理临时目录**：`/tmp`、`/var/tmp` 或 Agent 工作目录下的临时生成文件
- **清理缓存**：清理模型推理缓存、下载的中间文件、失败任务的残留文件
- **会话状态清理**：清理已过期/完成的会话上下文，释放内存

### 3. 数据库/存储维护
- **清理过期记录**：删除过期的任务历史、心跳记录、临时消息队列
- **碎片整理**：如果是 SQLite 等轻量数据库，定期 `VACUUM`
- **备份检查**：确认增量备份是否成功，清理过期备份（保留最近 N 个版本）

---

## 二、日常检查任务

### 1. 资源监控
| 检查项 | 阈值建议 | 异常处理 |
|--------|---------|---------|
| CPU 使用率 | 持续 >80% 超 10 分钟 | 记录 Top 进程，触发告警 |
| 内存使用 | 可用 <10% | 检查内存泄漏，考虑重启服务 |
| 磁盘空间 | 使用率 >85% | 紧急清理，扩容预警 |
| 网络连接 | 异常断连/延迟过高 | 检查上游服务状态 |

### 2. 服务健康检查
- **自检测 API**：调用 Agent 自身的 `/health` 或状态接口，确认响应正常
- **依赖服务检查**：数据库、消息队列、外部 API 的连通性测试
- **任务队列监控**：检查是否有**死信队列**（长期失败未重试的任务）

### 3. 模型/配置一致性检查
- **模型文件完整性**：校验模型权重文件的 MD5/SHA256（防止磁盘损坏导致异常）
- **配置漂移检测**：对比当前运行配置与版本控制中的基准配置，发现未授权修改
- **Token 余额/配额**：检查 API Key 余额、速率限制使用情况

---

## 三、安全与合规检查

- **异常访问日志**：检查是否有异常 IP、高频失败登录、未授权接口调用
- **敏感数据扫描**：确认日志/临时文件中未意外写入 API Key、密码等
- **权限审计**：检查 Agent 运行账户的权限是否最小化，有无新增高权限文件

---

## 四、建议的每日任务排程

```
00:00  日志归档 + 过期清理

00:15  异常日志扫描 + 告警摘要生成

```

这是一个关于任务管理方法论的问题，我来帮你梳理每日基本任务的布置原则：

---

## 核心原则：少即是多

**每天只布置 1-3 个核心任务**，而非列十几项待办。完成关键任务带来的成就感，远胜于忙碌却一事无成。

---

## 布置框架

### 1. 按能量曲线分配
| 时间段 | 适合的任务类型 |
|--------|-------------|
| 早晨（高能量） | 创造性工作、复杂决策、学习新内容 |
| 下午（中能量） | 沟通协调、常规事务、会议 |
| 晚上（低能量） | 复盘整理、简单收尾、准备次日计划 |

### 2. 任务分类法（推荐「3+2」模式）
- **3 个主线任务**：今天必须推进的工作/学习目标
- **2 个支线任务**：维护性事务（回复消息、整理文件等）

### 3. 每个任务要具体可执行
❌ "学英语" → ✅ "背 20 个雅思单词，完成 1 篇阅读精读"  
❌ "写报告" → ✅ "完成报告第二部分的数据分析，约 800 字"

---

## 每日布置流程建议

**前一晚或当天早晨花 5 分钟：**

1. **清空收集箱**：把脑子里所有想做的事写下来
2. **筛选优先级**：问自己"今天不做哪件事会后悔？"
3. **估算时间**：每个任务预留 1.5 倍预估时间
4. **预留缓冲**：只计划 80% 的时间，留 20% 应对突发

---

## 一个实用模板

```
今日主线：
□ 1. _____________（上午完成）
□ 2. _____________（下午完成）
□ 3. _____________（机动时间）

今日支线：
□ 回复 ____ 的邮件
□ 整理本周文档

备注：如遇干扰，优先保主线 1
```

---

## 关键提醒

- **完成比完美重要**：主线任务做到 80 分就停，避免陷入细节
- **晚间复盘**：记录"计划用时 vs 实际用时"，逐步校准自己的估算能力
- **周一看全局**：每周日/一布置本周方向，再拆到每天
